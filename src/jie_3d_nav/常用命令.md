本文档整理当前推荐的 `MID360 + FAST-LIVO2 + octomap 2D 投影 + ICP 重定位 + Nav2` 使用流程，按下面四部分阅读：

1. 介绍
2. 实车实时建图、保存 2D 地图、启动导航
3. 启动仿真
4. 常见问题排查

离线滤波参数测试与 bag 建图复盘流程见：

```text
/home/yangxuan/ros2_ws/src/offline_filter_and_bag_mapping.md
```

## 1. 介绍

### 1.1 整体链路

```text
MID360
-> livox_ros_driver2
-> FAST-LIVO2
-> /cloud_registered
-> octomap_server
-> /projected_map
-> map_saver_cli 保存 pgm/yaml
-> ICP 重定位发布 map -> odom
-> Nav2 使用 map -> odom -> livox_frame 导航
```

实车在线建图：

```text

```


```text

```

```text
mid360_nav_demo/launch/sim_relocalization_demo.launch.py
```

```text
mid360_nav_demo/scripts/octomap_tuning_gui.py
```

## 2. 实车实时建图、保存 2D 地图、启动导航

### 2.1 启动实时建图

这条命令会同时启动：

```text
octo_planner/launch/mid360_driver.launch.py
FAST-LIVO2-ROS2/launch/mapping_mid360_lio.launch.py
mid360_nav_demo/launch/octomap_mapping.launch.py
RViz（默认显示 /cloud_registered 和 /projected_map）
octomap_tuning_gui（可在线调高度参数并保存 2D 地图）
```

启动：

```bash
cd /home/yangxuan/ros2_ws/src
source /opt/ros/humble/setup.bash
source /home/yangxuan/ros2_ws/src/install/setup.bash

ros2 launch mid360_nav_demo online_mapping_demo.launch.py
```

如果要临时指定初始高度过滤参数：

```bash
ros2 launch mid360_nav_demo online_mapping_demo.launch.py \
  pointcloud_min_z:=0.05 \
  occupancy_min_z:=0.08 \
  pointcloud_max_z:=1.20 \
  occupancy_max_z:=1.20 \
  filter_speckles:=true
```

如果不想启动调参 GUI：

```bash
ros2 launch mid360_nav_demo online_mapping_demo.launch.py \
  launch_tuning_gui:=false
```

### 2.3 在线调参和保存 2D 地图

`online_mapping_demo.launch.py` 现在默认会启动 `MID360 Octomap Tuning` GUI。

GUI 支持：

```text
Refresh：读取 /octomap_server 当前参数
Apply Parameters：实时写入高度过滤参数
Apply + Reset Octomap：写参数后清空 octomap，适合观察新效果
Reset Only：只清空当前 octomap
Save PGM/YAML：保存当前 /projected_map
```

常用可调参数：

```text
sensor_max_range
pointcloud_min_z
pointcloud_max_z
occupancy_min_z
occupancy_max_z
filter_speckles
filter_ground
```

调参建议：

```text
远处回到近处后边线中间出现空白：适当增大 sensor_max_range
地面黑色杂点多：提高 pointcloud_min_z 和 occupancy_min_z
低矮障碍识别不到：降低 pointcloud_min_z 和 occupancy_min_z
高处结构干扰：降低 pointcloud_max_z 和 occupancy_max_z
孤立黑点多：保持 filter_speckles:=true
```

推荐起步值：

```bash
pointcloud_min_z:=0.05
occupancy_min_z:=0.08
pointcloud_max_z:=1.20
occupancy_max_z:=1.20
filter_speckles:=true
```

### 2.6 建图时录包，便于复盘和离线优化

如果这次实地数据后面还要用于：

```text
问题复盘
FAST-LIVO2 参数重调
octomap 2D 投影参数离线优化
重定位或导航问题复现
```

建议在建图时同步录一个必要话题集合的 bag。

推荐最小录制集合：

```text
/livox/lidar
/livox/imu
/cloud_registered
/aft_mapped_to_init
/tf
/tf_static
/projected_map
```

含义：

```text
/livox/lidar：原始雷达输入，离线重跑 FAST-LIVO2 的关键数据
/livox/imu：原始 IMU 输入
/cloud_registered：FAST-LIVO2 输出的配准点云，便于直接复盘 octomap 和导航输入
/aft_mapped_to_init：FAST-LIVO2 里程计输出
/tf、/tf_static：复盘坐标关系
/projected_map：当时实时生成的 2D 投影结果
```

推荐录包命令：

```bash
cd /home/yangxuan/ros2_ws/src
source /opt/ros/humble/setup.bash
source /home/yangxuan/ros2_ws/src/install/setup.bash

mkdir -p /home/yangxuan/ros2_ws/src/bags

ros2 bag record \
  -o /home/yangxuan/ros2_ws/src/bags/mid360_mapping_$(date +%Y%m%d_%H%M%S) \
  /livox/lidar \
  /livox/imu \
  /cloud_registered \
  /aft_mapped_to_init \
  /tf \
  /tf_static \
  /projected_map
```

如果还想一起复盘导航控制链路，也可以额外录：

```text
/cmd_vel
/ctrl_cmd
/io_cmd
/chassis_info_fb
/wheel_odom
```

对应命令：

```bash
ros2 bag record \
  -o /home/yangxuan/ros2_ws/src/bags/mid360_full_$(date +%Y%m%d_%H%M%S) \
  /livox/lidar \
  /livox/imu \
  /cloud_registered \
  /aft_mapped_to_init \
  /tf \
  /tf_static \
  /projected_map \
  /cmd_vel \
  /ctrl_cmd \
  /io_cmd \
  /chassis_info_fb \
  /wheel_odom
```

录包时建议：

```text
建图主流程一个终端
录包单独一个终端
保存 2D 地图后再结束录包
bag 文件名里带日期时间，方便后面对应地图和 PCD
```

录包结束后可先检查：

```bash
ros2 bag info /home/yangxuan/ros2_ws/src/bags/你的bag目录
```

回放时常用：

```bash
ros2 bag play /home/yangxuan/ros2_ws/src/bags/你的bag目录 --clock
```

如果要离线重跑依赖仿真时间的节点，记得对应 launch 加上：

```text
use_sim_time:=true
```



连底盘：

```bash

cd /home/yangxuan/ros2_ws/src
source /opt/ros/humble/setup.bash
source /home/yangxuan/ros2_ws/src/install/setup.bash

ros2 launch mid360_nav_demo online_nav_demo.launch.py \
  map:=/home/yangxuan/ros2_ws/src/mid360_nav_demo/maps/site1.yaml \

  launch_rviz:=true \
  launch_cmd_bridge:=true \
  launch_chassis:=true \
  bridge_publish_rate:=30.0
```

RViz 操作顺序：

```text
1. Fixed Frame 设为 map
2. 确认 /map 可见
3. 点 2D Pose Estimate 触发 ICP 重定位
4. 等 map -> odom 正常
5. 再发布 Nav2 Goal
```

导航检查命令：

```bash
ros2 topic echo /relocalization/status --once
ros2 run tf2_ros tf2_echo map odom
ros2 run tf2_ros tf2_echo odom livox_frame
ros2 lifecycle get /map_server
ros2 lifecycle get /controller_server
ros2 lifecycle get /planner_server
ros2 lifecycle get /bt_navigator
ros2 lifecycle get /velocity_smoother
ros2 topic hz /cmd_vel -w 5
ros2 topic hz /ctrl_cmd -w 10
```

这一节的 `online_nav_demo.launch.py` 属于原始导航入口，特点是：

```text
Nav2 /cmd_vel -> 原始 cmd_vel_to_ctrl_cmd_bridge -> /ctrl_cmd
不经过 agt_nav_console
不经过 safety_guard_node
不经过 obstacle_stop_node
```

使用建议：

```text
只想验证原始 MID360 + FAST-LIVO2 + Nav2 基本功能时，用 2.7 这一节
需要安全速度仲裁、障碍急停、Qt GUI 联调时，不要再用 2.7，改用 2.8
当前推荐的实车联调入口是 agt_nav_console/safe_online_nav_demo.launch.py
```

### 2.8 安全导航链路 + Qt GUI 联调

链路：

```text

-> Nav2 规划与控制

-> /agt/cmd_vel_nav
-> safety_guard_node
```

```text

-> /scan
-> obstacle_stop_node
-> /agt/obstacle_stop
-> safety_guard_node
```

## 4. 常见问题排查

### 4.1 `/projected_map` 看不到

先确认是否真的有实时发布：

```bash
ros2 topic info /projected_map -v
ros2 topic echo /projected_map --once --field info
```

如果 topic 有数据但 RViz 看不到，确认：

```text
Fixed Frame 是否为 odom
Map display 话题是否为 /projected_map
是否使用了带 Projected Map 的 RViz 配置
```

### 4.2 `/projected_map` 保存失败

优先使用当前保存 launch：

```bash
ros2 launch mid360_nav_demo save_projected_map.launch.py \
  map_prefix:=/home/yangxuan/ros2_ws/src/mid360_nav_demo/maps/site1
```

如果只是超时：

```bash
ros2 launch mid360_nav_demo save_projected_map.launch.py \
  map_prefix:=/home/yangxuan/ros2_ws/src/mid360_nav_demo/maps/site1 \
  save_map_timeout:=60.0
```

### 4.3 PGM 里地面黑色杂点很多

说明黑点已经被投进了 `/projected_map`，不是单纯 RViz 显示问题。

优先尝试：

```bash
pointcloud_min_z:=0.08
occupancy_min_z:=0.10
filter_speckles:=true
```

如果低矮障碍也被滤掉了，再适当降低：

```bash
pointcloud_min_z:=0.03
occupancy_min_z:=0.05
```

### 4.4 `/map` 或 `map` frame 不存在

检查：

```bash
ros2 lifecycle get /map_server
ros2 topic info /map -v
ros2 topic echo /relocalization/status --once
ros2 run tf2_ros tf2_echo map odom
```

如果 `/map` 有但 `map` frame 不存在，通常是 ICP 还没成功，没有发布 `map -> odom`。

### 4.5 Nav2 已启动但车不动

先看控制链路：

```bash
ros2 topic echo /cmd_vel --once
ros2 topic echo /ctrl_cmd --once
ros2 topic echo /io_cmd --once
ros2 topic echo /chassis_info_fb --once
```

### 4.9 当前实车联调已知限制

当前这套 `MID360 + FAST-LIVO2 + Nav2 + agt_nav_console + Ros_Qt5_Gui_App` 联调链路，先默认记住下面两点：

```text
1. 当前没有完整机器人描述/碰撞模型参与导航约束，局部规划和路径跟踪时存在贴边、擦碰或局部碰撞风险。
2. 当前规划与跟踪链路不以“后退完成路径跟踪”为目标设计，遇到需要倒车修正的场景时，通常不会主动后退完成轨迹。
3. 当前 Qt GUI 多点导航已经完成联调测试，但现场稳定可用规模暂时建议按“最多 7 个目标点”理解；继续增加目标点时，后续点位存在中途停下、不再继续执行的现象，具体原因还需要继续定位。
```

使用建议：

```text
首次实车联调时优先选开阔场地
先测短距离单点导航，不要直接测复杂多点任务
靠近墙边、立柱、转角和狭窄通道时要保留人工接管
不要假设车会像部分商用 AMR 一样自动倒车脱困或倒车贴路径
```

当前多点导航测试结论补充：

```text
1. 现阶段已经完成 Qt GUI 多点导航实测，任务链可以顺序触发多个 /goal_pose。
2. 当前稳定测试结果以 7 个目标点为上限更稳妥，超过后出现停车、不再继续切换下一目标点的情况。
3. 这类问题暂时怀疑与阿克曼底盘运动学约束、局部跟踪器在极限转向场景下的收敛性，以及“到点判定/任务切换条件”共同有关。
4. 一些较为极限的点到点路径当前无法稳定完成，尤其是在需要大曲率转向、贴边修正或较强姿态调整的场景中更明显。
5. 后续优化方向应优先考虑切换或重配路径规划器/控制器，并补强导航失败恢复逻辑，而不是只从 GUI 侧继续加任务点。
```

### 4.10 后续优化待办

建议按下面顺序推进：

```text
1. 先排查“多点任务中断”的直接触发条件：
   重点看 /goal_pose 是否继续发出、Nav2 是否仍处于 active、任务切换时的到点判定是否过严。

2. 重新评估局部控制器是否适合当前阿克曼底盘：
   当前 DWB 在一些大曲率、贴边和姿态约束较强的场景中不够稳，后续可优先评估更适合 Ackermann 运动学的控制器或参数组合。

3. 重新评估全局路径规划器：
   现有规划结果在部分极限场景下虽然能给出路径，但未必适合实际可跟踪性，后续可以考虑切换或重配全局规划器，提高路径可执行性。

4. 补强导航失败恢复逻辑：
   当前需要增加规划失败、控制失败、长时间原地停滞、目标不可达时的恢复策略，避免任务链直接停死。

5. 补充机器人描述与碰撞模型：
   需要把更真实的车体包络、前后悬和转弯约束纳入导航配置，降低贴边和擦碰风险。

6. 重新验证多点任务规模上限：
   在控制器、规划器和恢复行为调整后，再复测 7 点以上任务链，确认中断问题是否消失。
```
