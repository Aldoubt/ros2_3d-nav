# 导航测试流程

本文档用于 `2026-06-07` 现场导航测试准备与执行，目标是把建图、地图导出、重定位、单点/多点导航测试、录包和数据备份整理成一套可直接执行的流程。

## 测试目标

- 验证 `FAST-LIVO2 + 重定位 + Nav2 + Qt GUI + nav_test_recorder` 主链是否稳定
- 验证单点导航到点精度
- 验证多点导航路径合理性与执行稳定性
- 留存可复盘的地图、bag、指标和现场备注

### 环境准备

- 确认工作区已成功构建
- 确认 `Ros_Qt5_Gui_App` 可启动
- 确认 `nav_test_recorder` 可启动
- 确认录制目录有足够空间
- 确认使用的是正确地图版本和测试点版本

## 阶段 1：建图测试

### 目标

- 生成完整可用的点云地图
- 尽量保证闭环
- 保留用于后续重定位的原始点云地图

### 推荐扫描顺序

1. 先沿场地边缘扫一圈
2. 再扫描主行道和转弯区域
3. 补扫遮挡多、结构复杂或容易漂移的位置
4. 最后随机补扫遗漏区域

### 建图建议

- 优先保证闭环，不要只随机走
- 转角、狭窄通道、开阔区域都要覆盖
- 尽量避免只扫一条直线
- 建图过程中如果发现明显漂移，及时重来，不要带病进入导航阶段

### 建图后检查

- 地图是否覆盖完整
- 是否存在明显重影
- 是否存在大范围漂移
- 是否已经导出点云地图

## 阶段 2：导出导航地图

### 目标

- 导出 `pgm + yaml` 供 `Nav2` 使用
- 保留与之对应的点云地图供重定位使用

### 输出要求

- 一份 `pgm`
- 一份 `yaml`
- 一份用于重定位的 `.pcd`

### 检查项

- `yaml` 中分辨率是否合理
- `origin` 是否正常
- 地图方向是否正确
- RViz 中加载后是否与实际场地一致

## 阶段 3：规划测试点并保存版本

### 目标

- 生成单点测试点
- 生成多点测试路线
- 保存测试点版本，和地图版本绑定

### 建议规则

- 单点测试至少覆盖：
  - 近距离点
  - 长距离点
  - 转弯点
  - 狭窄区域点
- 多点测试至少覆盖：
  - 连续转弯
  - 往返
  - 长走廊

### 注意

- 先保存地图版本，再保存测试点版本
- 不要在测试中途改地图又继续沿用旧测试点

## 阶段 5：开始正式测试

### 人员撤离前

- 设好初始位姿
- 确认目标点发布链路正常
- 点击 recorder 的“开始录制”
- 点击 recorder 的 `start`
- 确认现场人员已撤离到安全位置

### 单点导航测试

每个点建议记录：

- 是否成功到点
- 到点时间
- 终点偏差
- 是否人工接管
- 是否障碍触发
- 是否出现明显路径异常

推荐操作顺序：

1. 在 Qt GUI 中输入目标点
2. 观察 RViz 中全局/局部路径
3. 观察机器人是否稳定前进
4. 到点后点击 `arrive`
5. 若失败点击 `fail`
6. 若人工接管点击 `takeover`
7. 若有突发情况用 `note` 追加说明

### 多点导航测试

重点关注：

- 路线是否合理
- 是否频繁重规划
- 转弯是否抖动
- 狭窄区域是否容易卡住
- 多点顺序切换是否稳定

建议每段都补一条备注，说明具体异常位置或现象。

## 阶段 6：测试结束后检查

### recorder 结束操作

完成当前轮测试后：

1. 点击 `arrive` 或 `fail`
2. 如有需要补 `note`
3. 点击“停止并保存”

### 检查输出目录

每轮测试完成后，确认目录内至少包含：

- `bag/`
- `metrics.csv`
- `events.csv`
- `metadata.yaml`

如果 `bag/` 不存在或 `metrics.csv` 只有表头，这轮测试应视为记录失败，需要备注原因。

## 阶段 7：结果分析

推荐测试结束后立即做一次快速分析：

```bash
cd /home/yangxuan/ros2_ws
source /opt/ros/humble/setup.bash
source /home/yangxuan/ros2_ws/install/setup.bash

ros2 run agt_nav_console analyze_nav_test /tmp/nav_tests/site1_nav_run_01
```

当前重点看：

- `path_length_m`
- `arrival_time_s`
- `max_speed_mps`
- `goal_error_m`

### 分析时的判断建议

- `goal_error_m` 很大：优先检查是否真的到点、是否手动提前点了 `arrive`
- `max_speed_mps` 接近 0：优先检查 `/cmd_vel_safe` 是否真实输出
- bag 太小：优先检查是否确实录到了 `/cloud_registered`

## 阶段 8：数据备份

每轮测试结束后建议立即备份：

- `bag/`
- `metrics.csv`
- `events.csv`
- `metadata.yaml`
- `pgm + yaml`
- `.pcd`
- 测试点配置
- 现场视频或照片

推荐至少保留两份：

- 本机工作目录
- 外部硬盘或另一个存储目录
