# 导航测试流程

本文档用于 `2026-06-07` 现场导航测试准备与执行，目标是把建图、地图导出、重定位、单点/多点导航测试、录包和数据备份整理成一套可直接执行的流程。

## 测试目标

- 验证 `FAST-LIVO2 + 重定位 + Nav2 + Qt GUI + nav_test_recorder` 主链是否稳定
- 验证单点导航到点精度
- 验证多点导航路径合理性与执行稳定性
- 留存可复盘的地图、bag、指标和现场备注

## 测试前准备

### 环境准备

- 确认工作区已成功构建
- 确认 `Ros_Qt5_Gui_App` 可启动
- 确认 `nav_test_recorder` 可启动
- 确认录制目录有足够空间
- 确认使用的是正确地图版本和测试点版本

建议先执行：

```bash
export WS_ROOT=/path/to/your/ros2_ws
export ROS_DISTRO=${ROS_DISTRO:-humble}
export MAP_PATH=${MAP_PATH:-$WS_ROOT/src/mid360_nav_demo/maps/example_site.yaml}
export GLOBAL_MAP_PCD=${GLOBAL_MAP_PCD:-$WS_ROOT/src/FAST-LIVO2-ROS2/Log/PCD/all_raw_points.pcd}
export ROS_QT5_GUI_APP_DIR=${ROS_QT5_GUI_APP_DIR:-$WS_ROOT/src/Ros_Qt5_Gui_App}
export NAV_TEST_OUTPUT_ROOT=${NAV_TEST_OUTPUT_ROOT:-$WS_ROOT/nav_tests}

cd "${WS_ROOT}"
source "/opt/ros/${ROS_DISTRO}/setup.bash"
source "${WS_ROOT}/install/setup.bash"

ros2 pkg executables agt_nav_console
```

### 需要留档的信息

每轮测试开始前建议先记下：

- 日期
- 场地名称
- 测试人员
- 地图版本
- 点云地图版本
- 测试点版本
- 车辆/底盘状态
- 备注

推荐命名方式：

- 地图：`example_site_map_v1`
- 点云地图：`example_site_pcd_v1`
- 测试点：`example_site_waypoints_v1`
- 导航测试：`example_site_nav_run_01`

## 阶段 1：建图测试

### 目标

- 生成完整可用的点云地图
- 尽量保证闭环
- 保留用于后续重定位的原始点云地图

### 推荐扫描顺序

1. 先沿场地边缘扫一圈
2. 再扫描主行道和转弯区域
3. 补扫遮挡多、结构复杂或容易漂移的位置
4. 最后随机补扫遗漏区域

### 建图建议

- 优先保证闭环，不要只随机走
- 转角、狭窄通道、开阔区域都要覆盖
- 尽量避免只扫一条直线
- 建图过程中如果发现明显漂移，及时重来，不要带病进入导航阶段

建议将建图阶段录制的 bag 保存到 `${WS_ROOT}/src/bags/` 目录，例如：

```bash
mkdir -p "${WS_ROOT}/src/bags"

ros2 bag record -o "${WS_ROOT}/src/bags/mid360_cartographer" \
  /livox/lidar \
  /livox/imu \
  /tf \
  /tf_static \
  /aft_mapped_to_init
```

### 建图后检查

- 地图是否覆盖完整
- 是否存在明显重影
- 是否存在大范围漂移
- 是否已经导出点云地图

## 阶段 2：导出导航地图

### 目标

- 导出 `pgm + yaml` 供 `Nav2` 使用
- 保留与之对应的点云地图供重定位使用

### 输出要求

- 一份 `pgm`
- 一份 `yaml`
- 一份用于重定位的 `.pcd`

### 检查项

- `yaml` 中分辨率是否合理
- `origin` 是否正常
- 地图方向是否正确
- RViz 中加载后是否与实际场地一致

## 阶段 3：规划测试点并保存版本

### 目标

- 生成单点测试点
- 生成多点测试路线
- 保存测试点版本，和地图版本绑定

### 建议规则

- 单点测试至少覆盖：
  - 近距离点
  - 长距离点
  - 转弯点
  - 狭窄区域点
- 多点测试至少覆盖：
  - 连续转弯
  - 往返
  - 长走廊

### 注意

- 先保存地图版本，再保存测试点版本
- 不要在测试中途改地图又继续沿用旧测试点

## 阶段 4：启动重定位与导航链

### 推荐终端分工

#### 终端 1：导航主链

```bash
  launch_rviz:=true \
  launch_chassis:=true \
  bridge_publish_rate:=30.0
```

#### 终端 2：Qt GUI

```bash
cd "${WS_ROOT}"
source "/opt/ros/${ROS_DISTRO}/setup.bash"
source "${WS_ROOT}/install/setup.bash"

"${WS_ROOT}/src/agt_nav_console/scripts/start_qt_gui.sh" \
  "${ROS_QT5_GUI_APP_DIR}"
```

#### 终端 3：录包与实验记录

```bash
cd "${WS_ROOT}"
source "/opt/ros/${ROS_DISTRO}/setup.bash"
source "${WS_ROOT}/install/setup.bash"

ros2 run agt_nav_console nav_test_recorder \
  --gui \
  --profile debug \
  --output-root "${NAV_TEST_OUTPUT_ROOT}" \

```

`debug` 预设当前会录制：

- `/aft_mapped_to_init`
- `/cmd_vel_safe`
- `/goal_pose`
- `/agt/obstacle_stop`
- `/cloud_registered`
- `/plan`
- `/local_plan`

#### 终端 4：辅助检查

```bash
cd "${WS_ROOT}"
source "/opt/ros/${ROS_DISTRO}/setup.bash"
source "${WS_ROOT}/install/setup.bash"

ros2 topic list | sort
ros2 node list | sort
```

### 启动后必须确认

- 地图已正常加载
- 重定位成功
- Qt GUI 可正常设点
- `/goal_pose` 可发布
- `/cmd_vel_safe` 有输出
- recorder 已进入待命界面
- recorder 显示的录制预设和话题列表正确

### TF / 定位检查

当前链路重点检查：

```text
map -> odom -> livox_frame
```

可用命令：

```bash
ros2 run tf2_ros tf2_echo odom livox_frame
ros2 run tf2_ros tf2_echo map odom
ros2 run tf2_ros tf2_echo map livox_frame
```

判断标准：

- `odom -> livox_frame` 正常更新
- `map -> odom` 已建立
- `map -> livox_frame` 稳定

## 阶段 5：开始正式测试

### 人员撤离前

- 设好初始位姿
- 确认目标点发布链路正常
- 点击 recorder 的“开始录制”
- 点击 recorder 的 `start`
- 确认现场人员已撤离到安全位置

### 单点导航测试

每个点建议记录：

- 是否成功到点
- 到点时间
- 终点偏差
- 是否人工接管
- 是否障碍触发
- 是否出现明显路径异常

推荐操作顺序：

1. 在 Qt GUI 中输入目标点
2. 观察 RViz 中全局/局部路径
3. 观察机器人是否稳定前进
4. 到点后点击 `arrive`
5. 若失败点击 `fail`
6. 若人工接管点击 `takeover`
7. 若有突发情况用 `note` 追加说明

### 多点导航测试

重点关注：

- 路线是否合理
- 是否频繁重规划
- 转弯是否抖动
- 狭窄区域是否容易卡住
- 多点顺序切换是否稳定

建议每段都补一条备注，说明具体异常位置或现象。

## 阶段 6：测试结束后检查

### recorder 结束操作

完成当前轮测试后：

1. 点击 `arrive` 或 `fail`
2. 如有需要补 `note`
3. 点击“停止并保存”

### 检查输出目录

每轮测试完成后，确认目录内至少包含：

- `bag/`
- `metrics.csv`
- `events.csv`
- `metadata.yaml`

如果 `bag/` 不存在或 `metrics.csv` 只有表头，这轮测试应视为记录失败，需要备注原因。

## 阶段 7：结果分析

推荐测试结束后立即做一次快速分析：

```bash
cd "${WS_ROOT}"
source "/opt/ros/${ROS_DISTRO}/setup.bash"
source "${WS_ROOT}/install/setup.bash"

ros2 run agt_nav_console analyze_nav_test "${NAV_TEST_OUTPUT_ROOT}/example_site_nav_run_01"
```

当前重点看：

- `path_length_m`
- `arrival_time_s`
- `max_speed_mps`
- `goal_error_m`

### 分析时的判断建议

- `goal_error_m` 很大：优先检查是否真的到点、是否手动提前点了 `arrive`
- `max_speed_mps` 接近 0：优先检查 `/cmd_vel_safe` 是否真实输出
- bag 太小：优先检查是否确实录到了 `/cloud_registered`

## 阶段 8：数据备份

每轮测试结束后建议立即备份：

- `bag/`
- `metrics.csv`
- `events.csv`
- `metadata.yaml`
- `pgm + yaml`
- `.pcd`
- 测试点配置
- 现场视频或照片

推荐至少保留两份：

- 本机工作目录
- 外部硬盘或另一个存储目录

## 总控 GUI 总体运行逻辑

如果现场不想再手工开很多终端，建议直接使用：

```bash
source "/opt/ros/${ROS_DISTRO}/setup.bash"
source "${WS_ROOT}/install/setup.bash"

ros2 run agt_nav_console nav_ops_console
```

这条命令本身不会立刻启动导航相关 ROS 节点，它先拉起一个 `tkinter` 写的总控上位机界面。这个界面的作用是把“建图、导图、Qt GUI、导航主链、记录器”几段流程串起来，按按钮再分别启动对应进程。

### 1. 入口与基础依赖

- 命令入口是 `agt_nav_console` 包安装出的脚本封装，最终执行 `scripts/nav_ops_console.py`
- 总控 GUI 启动前依赖：
  - `/opt/ros/${ROS_DISTRO}/setup.bash`
  - `${WS_ROOT}/install/setup.bash`
- GUI 自身不是 ROS 节点，它是一个进程管理器
- 它内部通过 `bash -lc "source ... && ros2 ..."` 的方式去启动各类 launch 和工具

### 2. 总控 GUI 管理的 4 类进程

界面里主要维护 4 个“受控进程槽位”：

- `mapping`
  - 建图模式
- `nav`
  - 导航主链
- `recorder`
  - 导航记录器
- `qt_gui`
  - `Ros_Qt5_Gui_App`

这些进程都不是嵌入在 GUI 内运行，而是由总控 GUI 用子进程方式拉起，并把标准输出汇总到界面的“运行日志”窗口中。

### 3. 建图模式按钮背后的逻辑

点击“启动建图模式”后，总控 GUI 实际执行的是：

```bash
ros2 launch mid360_nav_demo online_mapping_demo.launch.py \
  fast_livo_params:=<建图参数文件> \
  launch_rviz:=true \
  launch_tuning_gui:=true
```

这条链路会继续拉起：

- `octo_planner/launch/mid360_driver.launch.py`
  - MID360 驱动
- `fast_livo` 的 `mapping_mid360_lio.launch.py`
  - FAST-LIVO2 建图
- `mid360_nav_demo/launch/octomap_mapping.launch.py`
  - `octomap_server`
  - 将 `/cloud_registered` 投影成 `/projected_map`
- `rviz2`
- `mid360_nav_demo` 的 `octomap_tuning_gui.py`
  - 在线调 `octomap_server` 高度/范围参数

这一阶段的核心依赖关系是：

- MID360 驱动提供雷达数据
- FAST-LIVO2 输出 `/cloud_registered`
- `octomap_server` 订阅 `/cloud_registered`，输出 `/projected_map`
- 后续“保存 PGM/YAML”就是对 `/projected_map` 调 `map_saver_cli`

### 4. 保存地图与导出点云的逻辑

点击“保存 PGM/YAML”时，总控 GUI不会常驻拉新节点，而是一次性执行：

```bash
ros2 launch mid360_nav_demo save_projected_map.launch.py \
  map_prefix:=<你在界面里填的前缀>
```

它本质上调用 `nav2_map_server map_saver_cli`，从 `/projected_map` 保存：

- `<prefix>.pgm`
- `<prefix>.yaml`

点击“导出最新点云地图”时，不是重新建图，而是把当前 FAST-LIVO2 默认输出目录里的：

- `${WS_ROOT}/src/FAST-LIVO2-ROS2/Log/PCD/all_raw_points.pcd`

复制到你在界面中指定的位置，避免后续重复建图时旧结果被覆盖。

点击“回填导航路径”或在保存/导出后，GUI 会自动把：

- 2D 地图路径回填到“导航地图 YAML”
- 点云地图路径回填到“重定位点云地图”

这样后面进入导航时，不需要再手工把路径抄一遍。

### 5. 启动 Qt GUI 的逻辑

点击“启动 Qt GUI”后，总控 GUI 实际执行：

```bash
${WS_ROOT}/src/agt_nav_console/scripts/start_qt_gui.sh <Ros_Qt5_Gui_App 目录>
```

这个脚本会继续检查以下几类可执行入口是否存在：

- `build/start.sh`
- `build/ros_qt5_gui_app`
- `build/install/bin/start.sh`
- `install/linux/bin/start.sh`

也就是说，总控 GUI 不负责编译 `Ros_Qt5_Gui_App`，只负责定位并启动已经编译好的 GUI。

### 6. 启动导航模式的两种逻辑

总控 GUI 中“导航启动模式”有两个选项：

- `safe`
  - 推荐现场实测使用
- `raw`
  - 接近手工直接运行 `online_nav_demo.launch.py`

#### `raw` 模式

点击“启动导航模式”，若选择 `raw`，实际执行：

```bash
ros2 launch mid360_nav_demo online_nav_demo.launch.py \
  map:=<导航地图 YAML> \
  nav2_params_file:=<Nav2 参数文件> \
  global_map_pcd:=<重定位点云地图> \
  launch_rviz:=true \
  launch_cmd_bridge:=true \
  launch_chassis:=true \
  bridge_publish_rate:=30.0
```

它会拉起的主链包括：

- MID360 驱动
- FAST-LIVO2 重定位配置
- `icp_relocalizer_node`
- `Nav2`
- `cmd_vel_to_ctrl_cmd_bridge`
- `yhs_can_control_node`

这是“直接导航链”，没有额外叠加总控安全链。

#### `safe` 模式

若选择 `safe`，实际执行：

```bash
ros2 launch agt_nav_console safe_online_nav_demo.launch.py \
  map:=<导航地图 YAML> \
  nav2_params_file:=<Nav2 参数文件> \
  global_map_pcd:=<重定位点云地图> \
  launch_rviz:=true \
  launch_chassis:=true \
  bridge_publish_rate:=30.0
```

这条链会先包含 `online_nav_demo.launch.py` 的主链，但额外再叠加三层安全/桥接逻辑：

- `SetRemap(src="/cmd_vel", dst="/agt/cmd_vel_nav")`
  - 先把 Nav2 的速度输出改道
- `safety_guard_node`
  - 安全速度仲裁
- `obstacle_stop_demo.launch.py`
  - `pointcloud_to_laserscan`
  - `obstacle_stop_node`
- `cmd_vel_to_ctrl_cmd_bridge_safe`
  - 将 `/cmd_vel_safe` 转成底盘 `ctrl_cmd`

所以 `safe` 模式下的控制链可以理解为：

`Nav2 /cmd_vel`
-> `/agt/cmd_vel_nav`
-> 安全仲裁 / 障碍急停
-> `/cmd_vel_safe`
-> 底盘桥接
-> `/ctrl_cmd`

这也是为什么现场实测更推荐走 `safe`。

### 7. 动态障碍与膨胀半径的控制逻辑

总控 GUI 在启动导航前还会处理三类可选覆写：

- `disable_dynamic_obstacles`
  - 禁用动态障碍更新，仅保留静态地图
- `local_inflation_radius`
  - 覆写局部膨胀半径
- `global_inflation_radius`
  - 覆写全局膨胀半径

这些值会一路透传到导航 launch，最终由 `nav2_mid360.launch.py` 动态改写 Nav2 参数文件后生效。

因此总控 GUI 本质上不是自己改 Nav2，而是：

- 读取现有 Nav2 参数文件
- 给你看当前膨胀半径提示
- 在启动导航时把覆写参数传进 launch

### 8. 记录器按钮背后的逻辑

点击“启动记录器”后，总控 GUI 执行：

```bash
ros2 run agt_nav_console nav_test_recorder \
  --gui \
  --profile <lite|debug> \
  --output-root <记录输出目录> \
  --test-name <测试名称>
```

这会再拉起一个独立的记录器 GUI，用来：

- 启停 `ros2 bag record`
- 记录 `start / arrive / fail / obstacle / takeover / note`
- 监控 CPU / 内存 / 磁盘
- 输出 `bag/`、`metrics.csv`、`events.csv`、`metadata.yaml`

点击“一键启动导航+记录”时，总控 GUI 的逻辑是：

1. 先启动导航主链
2. 延迟约 `1.5s`
3. 再启动记录器

这样做是为了尽量减少记录器早于主链启动时出现的空窗期。

### 9. 总控 GUI 推荐使用顺序

推荐按下面顺序使用总控 GUI：

1. 启动 `nav_ops_console`
2. 在“阶段 1”里启动建图模式
3. 调整 `octomap` 参数，确认 `/projected_map` 正常
4. 保存 `PGM/YAML`
5. 导出最新 `PCD`
6. 回填导航路径，确认导航 YAML 和 PCD 路径正确
7. 启动 `Qt GUI`
8. 选择 `safe` 模式启动导航主链
9. 启动记录器，或直接点“一键启动导航+记录”
10. 在 `Qt GUI` 中设初始位姿和目标点

### 10. 使用这套总控 GUI 时要注意

- 它负责“串流程”和“拉进程”，不替代每个子系统本身
- `Ros_Qt5_Gui_App` 必须已经先编译好
- 建图导出的 PCD 仍然来源于 FAST-LIVO2 默认 `Log/PCD` 目录
- 关闭总控 GUI 时，如果子进程还在运行，它会先询问是否停止全部进程
- “停止全部”发送的是中断信号，不是强杀；如果某个子进程卡死，还需要单独排查

### 12. 总控 GUI 时序图

更贴近现场操作顺序的理解可以看下面这版：

```text
1. 启动 nav_ops_console
   |
   +--> 只出现总控 GUI，本身还没有启动建图/导航/记录器

2. 点击“启动建图模式”
   |
   +--> MID360 驱动上线
   +--> FAST-LIVO2 开始建图
   +--> octomap_server 生成 /projected_map
   \--> RViz / 调参 GUI 可用于观察与调参

3. 点击“保存 PGM/YAML”
   |
   \--> 从 /projected_map 保存 2D 地图

4. 点击“导出最新点云地图”
   |
   \--> 从 FAST-LIVO2 默认输出目录复制最新 all_raw_points.pcd

5. 点击“回填导航路径”
   |
   \--> 把上一步生成的 YAML / PCD 自动填到导航区输入框

6. 点击“启动 Qt GUI”
   |
   \--> 拉起 Ros_Qt5_Gui_App，准备设初始位姿和目标点

7. 点击“启动导航模式”
   |
   +--> raw:
   |    直接启动 online_nav_demo 主链
   |
   \--> safe:
        在 online_nav_demo 主链外，再叠加
        - 安全仲裁
        - 障碍急停
        - safe cmd bridge

8. 点击“启动记录器”或“一键启动导航+记录”
   |
   \--> 拉起 nav_test_recorder GUI，开始录 bag 和事件

9. 在 Qt GUI 中发目标点
   |
   \--> Nav2 规划 -> 安全链处理 -> 底盘控制

10. 测试结束
    |
    +--> 记录器保存 bag / metrics / events / metadata
    \--> 总控 GUI 可分别停止 nav / recorder / qt_gui / mapping
```
